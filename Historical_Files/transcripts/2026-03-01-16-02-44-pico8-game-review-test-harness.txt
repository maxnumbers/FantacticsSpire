PICO-8's tight constraints (128\u00d7128 screen, 16 colors, 8192 tokens of Lua) make it an ideal target for AI game generation \u2014 the solution space is small enough to produce polished results.\n\n## Before You Start\n\nRead `references/pico8-api.md` for the full API reference, `.p8` file format spec, sprite encoding, and color palette. You will need it for every cart you generate.\n\n## The .p8 File Format\n\nA `.p8` file is a plain-text cartridge. It has labeled sections that you write directly:\n\n```\npico-8 cartridge // http://www.pico-8.com/version 42\nversion 42\n__lua__\n-- all game code here\n\n__gfx__\n-- 128 lines of 128 hex chars (sprite sheet, 8192 bytes)\n\n__gff__\n-- 2 lines of 256 hex chars (sprite flags)\n\n__map__\n-- 32 lines of 256 hex chars (map data, upper 32 rows)\n\n__sfx__\n-- 64 lines of sound effect data\n\n__music__\n-- 64 lines of music pattern data\n\n```\n\nEach hex character in `__gfx__` is one pixel (0-f = the 16 PICO-8 colors). Sprites are 8\u00d78 pixels, laid out left-to-right, top-to-bottom on a 128\u00d7128 sheet (16 sprites per row, 16 rows = 256 sprites). The lower half of the sprite sheet (sprites 128-255) shares memory with the lower half of the map.\n\n## Game Generation Workflow\n\n### Step 1: Understand the Request\n\nIdentify the genre, core mechanic, and scope. Ask clarifying questions only if the request is genuinely ambiguous. For straightforward requests (\"make a platformer\", \"make snake\"), just build it.\n\nThink about:\n- What genre? (platformer, shooter, puzzle, RPG, arcade, etc.)\n- What's the core loop? (jump/collect, shoot/dodge, match/clear, explore/fight)\n- What entities are needed? (player, enemies, pickups, projectiles, tiles)\n- What UI is needed? (score, lives, health bar, title screen, game over)\n\n### Step 2: Design the Sprite Sheet\n\nThis is critical. You must generate valid `__gfx__` hex data. Each row is 128 hex characters representing 128 pixels across the full sprite sheet width.\n\nPlan your sprite layout:\n- Sprites 0-15: Row 0 (player, player animation frames)\n- Sprites 16-31: Row 1 (enemies, enemy variants)\n- Sprites 32-47: Row 2 (pickups, items, projectiles)\n- Sprites 48-63: Row 3 (terrain/tiles \u2014 walls, ground, platforms)\n- Sprites 64+: Additional as needed\n\nDesign sprites by hand in hex. Each sprite is 8 pixels wide, so 2 sprites = 16 hex chars. Think about what shapes will read well at 8\u00d78.\n\n**Sprite design principles at 8\u00d78:**\n- Use 2-3 colors max per sprite for clarity\n- Outline with a dark color (1 or 0) for readability\n- Leave at least 1px padding from edges when possible\n- Symmetry reads well at low resolution\n- Animate with 2-frame alternation for simplicity\n\n**Color palette reference (hex digit \u2192 color):**\n```\n0: black       4: brown       8: red         c: light_gray\n1: dark_blue   5: dark_gray   9: orange      d: light_peach\n2: dark_purple 6: light_gray  a: yellow      e: pink\n3: dark_green  7: white       b: green       f: peach\n```\n\n### Step 3: Write the Lua Code\n\nStructure every game with this skeleton:\n\n```lua\n-- [game title]\n-- by claude\n\n-- game states\nfunction _init()\n  -- initialize game state variables\n  -- set up player, enemies, level data\n  gamestate=\"title\"\nend\n\nfunction _update60()\n  if gamestate==\"title\" then\n    update_title()\n  elseif gamestate==\"game\" then\n    update_game()\n  elseif gamestate==\"gameover\" then\n    update_gameover()\n  end\nend\n\nfunction _draw()\n  cls()\n  if gamestate==\"title\" then\n    draw_title()\n  elseif gamestate==\"game\" then\n    draw_game()\n  elseif gamestate==\"gameover\" then\n    draw_gameover()\n  end\nend\n```\n\n**Code patterns to follow:**\n\n- **Input:** Use `btn(0)`\u2013`btn(5)` for \u2b05\ufe0f\u27a1\ufe0f\u2b06\ufe0f\u2b07\ufe0f\ud83c\udd7e\ufe0f\u274e. Use `btnp()` for single-press actions (menu select, jump initiation).\n- **Collision (AABB):** `function collide(a,b) return a.x<b.x+b.w and a.x+a.w>b.x and a.y<b.y+b.h and a.y+a.h>b.y end`\n- **Map collision:** Check sprite flags with `fget(mget(tile_x, tile_y), flag_num)` to test if a tile is solid.\n- **Object pools:** Use tables as arrays: `enemies={}` \u2192 `add(enemies, {x=64,y=64,spd=1})` \u2192 iterate with `for e in all(enemies) do`.\n- **Removing objects:** Use `del(enemies, e)` inside iteration, PICO-8 handles this safely.\n- **Animation:** `spr(base_frame + flr(t/anim_speed) % num_frames, x, y)`\n- **Screen shake:** Offset camera with `camera(rnd(n)-n/2, rnd(n)-n/2)` then reset with `camera()`.\n- **Particles:** Simple table of `{x,y,dx,dy,life,col}` updated each frame.\n- **Coroutines:** Use for cutscenes, wave spawning, timed events.\n\n**Token conservation tips:**\n- Combine similar update logic for different entity types\n- Use short variable names for frequently accessed values\n- Prefer `if (cond) stmt` single-line form\n- Use `foreach(tbl, fn)` or `for v in all(tbl)` over index loops when you don't need the index\n- Reuse functions across game states where possible\n- Inline small computations rather than creating helper functions for one-off use\n\n### Step 4: Design Sound Effects (Optional but Recommended)\n\nThe `__sfx__` section contains 64 sound effects.
